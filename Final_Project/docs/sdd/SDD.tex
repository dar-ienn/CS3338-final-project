\section{Introduction}

\subsection{Purpose}

This Software Design Document (SDD) describes the architecture and internal system design for the LA City Sidewalk Assessment simulation project. The system is designed to support rover-based sidewalk inspection using LiDAR and point-cloud processing workflows.

\subsection{System Overview}

The system collects sidewalk scan data, processes LiDAR point clouds into orthoimages and elevation maps, uses a U-Net machine-learning model to identify sidewalk joints and cracks, calculates displacement measurements, and exports results for review.

\section{System Architecture}

The high-level system pipeline is:

Rover or iPhone LiDAR Scan $\rightarrow$ Raw Point Cloud $\rightarrow$ PointCloud2OrthoImage $\rightarrow$ Orthoimage and DEM/Elevation Image $\rightarrow$ U-Net Segmentation $\rightarrow$ Displacement Computation $\rightarrow$ CSV and Visualization Output $\rightarrow$ Reviewer Dashboard

\subsection{Snapshot 2 Data Processing Pipeline}

Snapshot 2 expands the system architecture by focusing on the data-processing and displacement-analysis workflow. The system processes LiDAR point-cloud scans into orthoimages and DEM/elevation images that are later used for segmentation and displacement calculation.

The updated Snapshot 2 workflow is:

\begin{itemize}
    \item Import raw LAS or PLY point-cloud files
    \item Align and normalize point-cloud data
    \item Generate orthoimages and DEM/elevation maps
    \item Process orthoimages using the U-Net segmentation model
    \item Generate segmentation masks identifying sidewalk joints and cracks
    \item Combine segmentation masks with DEM/elevation data
    \item Calculate vertical displacement measurements
    \item Export CSV reports and visualization outputs
\end{itemize}

The Snapshot 2 workflow improves the system by organizing the full processing pipeline into modular stages that can be validated independently during testing.

\subsection{Snapshot 3 Navigation Workflow}

Snapshot 3 expands the architecture by introducing simulated semi-autonomous navigation features for rover movement and obstacle management.

The updated navigation workflow includes:

\begin{itemize}
    \item GPS waypoint queue processing
    \item Sequential waypoint navigation
    \item Obstacle detection and avoidance
    \item Telemetry and battery monitoring
    \item Manual override controls
    \item Safe-stop behavior during navigation failures
\end{itemize}

The Snapshot 3 navigation architecture separates rover movement systems from the displacement-analysis workflow while still allowing telemetry and navigation logs to be stored alongside scan data.

\section{Module Breakdown}

\subsection{Data Acquisition Module}

\textbf{Purpose:}

The Data Acquisition Module is responsible for collecting sidewalk scan data using rover-mounted sensors, LiDAR-enabled devices, and GPS systems during field operations.

\textbf{Inputs:}

\begin{itemize}
    \item LiDAR scans
    \item GPS coordinates
    \item Sensor telemetry
    \item User scan commands
\end{itemize}

\textbf{Outputs:}

\begin{itemize}
    \item Raw point-cloud files
    \item Scan metadata
    \item Timestamped field data
\end{itemize}

\textbf{Dependencies:}

\begin{itemize}
    \item LiDAR-enabled iPhone or scanning device
    \item Rover hardware
    \item GPS module
\end{itemize}

\textbf{Failure Cases:}

\begin{itemize}
    \item Incomplete or corrupted scan
    \item Missing GPS data
    \item Unstable sensor readings
\end{itemize}

\subsection{Point-Cloud Processing Module}

\textbf{Purpose:}

The Point-Cloud Processing Module converts raw point-cloud scan data into orthoimages and DEM/elevation images that can be used by the machine-learning and displacement systems.

\textbf{Inputs:}

\begin{itemize}
    \item LAS point-cloud files
    \item PLY point-cloud files
\end{itemize}

\textbf{Outputs:}

\begin{itemize}
    \item Orthoimages
    \item DEM/elevation images
    \item Processed scan data
\end{itemize}

\textbf{Dependencies:}

\begin{itemize}
    \item PointCloud2OrthoImage processing tools
    \item Open3D
    \item Laspy
    \item NumPy
\end{itemize}

\textbf{Failure Cases:}

\begin{itemize}
    \item Corrupted point-cloud file
    \item Misaligned DEM/elevation output
    \item Excessive scan noise
    \item Unsupported file format
\end{itemize}

\textbf{Snapshot 2 Processing Additions:}

Snapshot 2 introduces a more detailed point-cloud processing workflow for displacement analysis.

The processing pipeline includes:

\begin{itemize}
    \item Point-cloud import and validation
    \item Alignment of sidewalk scans to a normalized coordinate system
    \item Noise reduction and filtering
    \item DEM/elevation image generation
    \item Orthoimage generation
    \item Output validation for downstream U-Net processing
\end{itemize}

\subsection{U-Net Segmentation Module}

\textbf{Purpose:}

The U-Net Segmentation Module identifies sidewalk joints, cracks, and slab boundaries from orthoimages.

\textbf{Inputs:}

\begin{itemize}
    \item Orthoimages generated from point-cloud data
    \item Trained U-Net model weights
\end{itemize}

\textbf{Outputs:}

\begin{itemize}
    \item Binary segmentation masks
    \item Predicted crack and joint locations
\end{itemize}

\textbf{Dependencies:}

\begin{itemize}
    \item Python
    \item TensorFlow/Keras
    \item OpenCV
    \item NumPy
\end{itemize}

\textbf{Failure Cases:}

\begin{itemize}
    \item Model fails to detect small cracks or slab joints
    \item False positive crack detections
    \item Input image resolution does not match expected model format
    \item Segmentation mask does not align with DEM/elevation image
\end{itemize}

\textbf{Snapshot 2 Segmentation Additions:}

Snapshot 2 improves the integration between the U-Net segmentation system and the displacement-analysis workflow.

The updated segmentation workflow includes:

\begin{itemize}
    \item Validation of orthoimage input dimensions
    \item Processing of normalized orthoimages
    \item Generation of binary segmentation masks
    \item Alignment verification between masks and DEM/elevation images
    \item Export of segmentation outputs for displacement analysis
\end{itemize}

\subsection{Displacement Computation Module}

\textbf{Purpose:}

The Displacement Computation Module calculates sidewalk displacement values using the DEM/elevation image and the segmentation mask produced by the U-Net module.

\textbf{Inputs:}

\begin{itemize}
    \item DEM/elevation images
    \item Segmentation masks
    \item Scan metadata
\end{itemize}

\textbf{Outputs:}

\begin{itemize}
    \item CSV files containing displacement measurements
    \item Visualization images showing detected sidewalk defects
    \item Measurement summaries for review
\end{itemize}

\textbf{Dependencies:}

\begin{itemize}
    \item Python
    \item NumPy
    \item Pandas
    \item SciPy
    \item Matplotlib
\end{itemize}

\textbf{Failure Cases:}

\begin{itemize}
    \item DEM/elevation image and segmentation mask do not align
    \item Missing left or right slab edge data
    \item Displacement value is unrealistic due to noisy scan data
    \item Output CSV file is incomplete or incorrectly formatted
\end{itemize}

\textbf{Snapshot 2 Displacement Additions:}

Snapshot 2 expands the displacement-analysis workflow by improving the relationship between segmentation masks and DEM/elevation data.

The updated displacement workflow includes:

\begin{itemize}
    \item Validation of DEM/elevation image alignment
    \item Detection of sidewalk slab boundaries
    \item Extraction of elevation differences across detected joints
    \item Calculation of vertical displacement measurements
    \item Export of CSV displacement reports
    \item Generation of visualization outputs for reviewers
\end{itemize}

The Semi-Autonomous Navigation Module simulates rover navigation features such as GPS waypoint navigation, obstacle detection, telemetry logging, manual override controls, and safe-stop behavior. Snapshot 3 focuses on expanding the navigation architecture and documenting how navigation systems interact with the rover workflow.

\textbf{Inputs:}

\begin{itemize}
    \item GPS waypoint coordinates
    \item Obstacle detection events
    \item Manual operator commands
    \item Telemetry and battery data
    \item Rover movement status
\end{itemize}

\textbf{Outputs:}

\begin{itemize}
    \item Navigation commands
    \item Telemetry logs
    \item Safe-stop events
    \item Navigation status updates
    \item Waypoint progress updates
\end{itemize}

\textbf{Dependencies:}

\begin{itemize}
    \item ROS or ROS Bridge
    \item GPS parsing libraries
    \item Obstacle detection sensors
    \item Telemetry logging services
    \item Rover control hardware
\end{itemize}

\textbf{Failure Cases:}

\begin{itemize}
    \item GPS waypoint drift
    \item Rover loses communication with operator
    \item False obstacle detection event
    \item Telemetry logging interruption
    \item Rover navigation path becomes blocked
\end{itemize}

\textbf{Snapshot 3 Navigation Additions:}

Snapshot 3 expands the system architecture by introducing simulated semi-autonomous navigation support. The navigation workflow allows the rover to process waypoint-based movement instructions while monitoring for obstacles and maintaining telemetry logs.

The updated Snapshot 3 navigation workflow includes:

\begin{itemize}
    \item GPS waypoint queue management
    \item Sequential waypoint execution
    \item Obstacle detection and avoidance events
    \item Manual override support
    \item Safe-stop behavior when navigation issues occur
    \item Telemetry logging and battery monitoring
    \item Navigation status reporting
\end{itemize}

The Snapshot 3 navigation architecture is designed to separate navigation processing from the displacement-analysis workflow while still allowing telemetry and scan data to be stored together for reviewer analysis.

\section{Interfaces}

\subsection{User Interfaces}

The system provides a browser-based control and monitoring interface that allows rover operators and reviewers to interact with the simulated sidewalk assessment system.

\subsection{Hardware Interfaces}

The system interfaces with rover hardware components including LiDAR-enabled mobile devices, GPS modules, obstacle detection sensors, and Raspberry Pi-based rover control hardware.

\subsection{Software Interfaces}

The system interfaces with TensorFlow/Keras, OpenCV, ROS or ROS Bridge, Docker services, GitHub, Jira, and TestRail.

\subsection{Communication Interfaces}

The system supports communication between rover hardware, processing workstations, storage systems, and development tools through simulated local network and cloud-based workflows.

\section{Design Constraints}

The system depends on LiDAR scan quality, point-cloud alignment accuracy, machine-learning model consistency, and compatible software dependencies such as TensorFlow, OpenCV, Laspy, and Open3D.

\section{Testing and Verification Strategy}

The system will be validated using a combination of manual testing, workflow verification, documentation review, and TestRail-generated reports.

Testing activities include:

\begin{itemize}
    \item Verification that required repository files and folders exist
    \item Validation of workflow diagram accuracy
    \item Review of SRS and SDD consistency
    \item Verification of point-cloud processing workflow documentation
    \item Validation of U-Net segmentation input and output descriptions
    \item Verification of displacement computation outputs and CSV report structure
    \item Review of navigation safety requirements and telemetry logging
    \item Validation of Docker configuration and dependency documentation
\end{itemize}

Testing evidence and exported reports will be maintained in the TestRail documentation folder throughout all project snapshots.
